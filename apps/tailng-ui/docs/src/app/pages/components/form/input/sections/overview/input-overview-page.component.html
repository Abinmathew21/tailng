Input

Inputs in TailNG are built as headless primitives that attach to native form controls and expose consistent slot markers and state hooks for styling.

You can use them in two ways:

  1. Direct attachment: tngInput on an <input> or <textarea>
  2. Grouped composition: tngInputGroup with optional leading/trailing content using slot directives

What you get

  • Native-first behavior
    • Uses real <input> and <textarea>.
    • Preserves standard browser behavior (typing, composition events, Enter submit, native validation).
  • Styling contract
    • Styling is driven by data-slot markers and state data attributes.
    • Group-level spacing between leading/trailing adornments and the control is exposed via --tng-input-inline-gap.
    • No required classes.
    • No DOM/tag assumptions beyond slot names.
  • Accessibility pass-through
    • Supports aria-label, aria-labelledby, aria-describedby, aria-required, aria-invalid.
    • aria-invalid is emitted only when invalid.

Accessibility guidance

  • Mark decorative leading/trailing content as aria-hidden="true".
  • Projected interactive elements (for example trailing clear button) are valid tab stops by design.

Testing notes

  • Use [ariaInvalid]="true" for stable invalid-state tests in jsdom.
  • Do not rely solely on :invalid unless test runtime supports it.
  • Group warning checks are dev-mode only and run asynchronously (microtask timing).
