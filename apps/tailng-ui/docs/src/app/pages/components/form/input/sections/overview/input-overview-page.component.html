Input

tngInput is the native input primitive, <tng-input> is the simple self-rendering component, and <tng-form-field> is the shell you use when the field needs prefixes, suffixes, or other projected content.

Use the simple component for the plain case, use tng-form-field for adorned fields, or use the primitives directly when you need full control over the markup.

  1. Simple component: use <tng-input> for a standard single input field
  2. Form field shell: use <tng-form-field> with a projected <input tngInput> when you need adornments
  3. Headless composition: use tngInputGroup when you want the raw primitive contract

What you get

  • Native-first behavior
    • <tng-input> still renders a real native <input>.
    • Preserves standard browser behavior (typing, composition events, Enter submit, native validation).
  • Styling contract
    • Styling is driven by data-slot markers and state data attributes.
    • Group-level spacing between leading/trailing adornments and the control is exposed via --tng-input-inline-gap.
    • No required classes.
    • No DOM/tag assumptions beyond slot names.
  • Accessibility pass-through
    • Supports aria-label, aria-labelledby, aria-describedby, aria-required, aria-invalid.
    • aria-invalid is emitted only when invalid.

Installation

Import only the primitives you need.

Basic usage

1) Simple component usage

<tng-input> renders its own native <input> and is the default component API for a plain field.

2) Form field shell for prefixes and suffixes

Use <tng-form-field> when the field needs projected adornments or actions.

3) Direct primitive attachment

Use this when you do not need wrapper chrome or projected slot content.

Structure

<tng-input> creates its own native control. <tng-form-field> wraps a projected <input tngInput> so you can add prefixes, suffixes, and custom trailing actions without losing the primitive contract.

Detailed API bindings are documented in the API tab. Styling slot/state contract is documented in the Styling tab. Textarea is documented on the dedicated Textarea page.

Accessibility guidance

  • Mark decorative leading/trailing content as aria-hidden="true".
  • Projected interactive elements (for example trailing clear button) are valid tab stops by design.

Testing notes

  • Use [ariaInvalid]="true" for stable invalid-state tests in jsdom.
  • Do not rely solely on :invalid unless test runtime supports it.
  • Group warning checks are dev-mode only and run asynchronously (microtask timing).
